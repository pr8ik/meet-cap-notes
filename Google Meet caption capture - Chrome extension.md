# Building a real-time Google Meet caption capture Chrome extension

**The most reliable approach to capturing Google Meet captions in 2025–2026 combines a MV3 content script using MutationObserver with ARIA-based selectors (`[aria-live]`) rather than obfuscated class names, periodic storage flushes to `chrome.storage.local`, and a service worker proxy for Claude API summarization.** Google Meet's DOM uses minified class names generated by Google's Closure/Wiz framework that change without warning — every production extension has broken at least once from selector changes. The open-source ecosystem offers battle-tested reference implementations, particularly TranscripTonic (171+ stars, actively maintained through 2026) and Recall.ai's open-source chrome extension with TypeScript caption scraping. This guide consolidates findings from those projects, Chrome extension documentation, and developer community experience into actionable technical guidance.

---

## Google Meet's caption DOM is volatile but follows a consistent pattern

Google Meet renders captions into a **live-updating `<div>` with `aria-live="polite"`** — an ARIA live region designed for screen readers that doubles as the most stable selector target for extensions. The internal structure as of early 2026 looks like this:

```html
<div aria-live="polite" role="region">
  <div class="iOzk7e">              <!-- Caption entry wrapper -->
    <div class="TBMuR">             <!-- Inner container -->
      <div class="NWpY1d">Speaker Name</div>        <!-- Speaker badge -->
      <span class="bh44bd">Caption text updates here</span>  <!-- Caption text -->
    </div>
  </div>
</div>
```

The critical selectors observed across multiple open-source projects are:

| Purpose | Stable selector | Fragile selector | Notes |
|---------|----------------|------------------|-------|
| **Caption live region** | `[aria-live]`, `div[aria-live="polite"]` | — | Most resilient; accessibility-mandated |
| **Speaker name badge** | — | `.NWpY1d`, `.xoMHSc` | Obfuscated; changes with Meet updates |
| **Caption text** | — | `.Mz6pEf`, `.bh44bd`, `.iTTPOb` | Multiple classes observed across versions |
| **Caption wrapper** | — | `.iOzk7e` | Per-speaker caption block |
| **Captions toggle** | `button[aria-label*="captions"]` | — | Keyboard shortcut `Shift+C` as fallback |
| **Leave call** | `button[aria-label*="Leave call"]` | — | Meeting end detection |
| **Participant images** | `div.U04fid > img[title]` | — | Name via `title` attribute |

**All class-name selectors (`NWpY1d`, `xoMHSc`, `iTTPOb`, `bh44bd`) are inherently fragile.** Google regenerates these at build time. The ARIA-based selectors (`[aria-live]`, `aria-label` on buttons) persist because Google must maintain accessibility compliance. Recall.ai's production documentation explicitly states: "A minor frontend change (e.g., class name, aria role) can break your scraper." The best mitigation is centralizing all selectors in a single constants file and using fallback chains.

The DOM updates in real-time as speech recognition processes audio — **text streams in word-by-word**, meaning a `<span>` might update from "Hello" → "Hello how" → "Hello how are you" within a single caption element. When a speaker finishes or a new speaker starts, Meet adds a new child `<div>` to the caption region and eventually removes stale ones. As of February 2025, Google added **caption history scrollback** (last 30 minutes), which means the DOM may retain more caption nodes than before.

## MutationObserver configuration and caption extraction code

The recommended MutationObserver configuration watches for both new caption elements and in-place text updates. The most battle-tested pattern comes from Recall.ai's open-source Google Meet bot (updated February 2026):

```javascript
const badgeSel = ".NWpY1d, .xoMHSc";
let lastSpeaker = "Unknown Speaker";

const getSpeaker = (node) => {
  const badge = node.querySelector(badgeSel);
  return badge?.textContent?.trim() || lastSpeaker;
};

const getText = (node) => {
  const clone = node.cloneNode(true);
  clone.querySelectorAll(badgeSel).forEach(el => el.remove());
  return clone.textContent?.trim() ?? "";
};

const processCaption = (node) => {
  const text = getText(node);
  const speaker = getSpeaker(node);
  if (text && text.toLowerCase() !== speaker.toLowerCase()) {
    handleNewCaption(speaker, text);
    lastSpeaker = speaker;
  }
};

new MutationObserver((mutations) => {
  for (const m of mutations) {
    // New caption elements added
    m.addedNodes.forEach(n => {
      if (n instanceof HTMLElement) processCaption(n);
    });
    // Live text edits within existing elements
    if (m.type === "characterData" && m.target?.parentElement) {
      processCaption(m.target.parentElement);
    }
  }
}).observe(document.body, {
  childList: true,      // New caption divs appearing
  characterData: true,  // Word-by-word text updates
  subtree: true         // Deep DOM observation required
});
```

**Three observer options work together**: `childList` catches new caption `<div>` elements when a speaker starts talking; `characterData` catches incremental text updates as words stream in; `subtree` is mandatory because captions nest deep within the DOM tree. The observer targets `document.body` because the caption container is dynamically created and destroyed — narrowing the target to the caption container alone risks missing it if it's re-created.

For **duplicate detection**, track active segments per speaker and only record when text has meaningfully changed:

```javascript
const activeSegments = new Map();
function handleNewCaption(speaker, caption) {
  const existing = activeSegments.get(speaker);
  if (!existing) {
    activeSegments.set(speaker, { speaker, text: caption, timestamp: Date.now() });
  } else if (caption.startsWith(existing.text) || caption.length > existing.text.length + 5) {
    existing.text = caption;  // Growing text — update in place
  } else {
    flushSegment(existing);   // New sentence — flush old, start new
    activeSegments.set(speaker, { speaker, text: caption, timestamp: Date.now() });
  }
}
```

Filter system messages that pollute the transcript: `/you left the meeting|return to home screen|leave call|feedback|audio and video|learn more/i`.

## Manifest V3 architecture optimized for long-running meetings

The recommended `manifest.json` for a Meet caption extension:

```json
{
  "manifest_version": 3,
  "name": "Meet Transcript",
  "version": "1.0.0",
  "permissions": ["storage", "unlimitedStorage", "activeTab", "downloads"],
  "host_permissions": [
    "https://meet.google.com/*",
    "https://api.anthropic.com/*"
  ],
  "background": { "service_worker": "service-worker.js", "type": "module" },
  "content_scripts": [{
    "matches": ["https://meet.google.com/*"],
    "js": ["content-script.js"],
    "run_at": "document_idle",
    "world": "ISOLATED"
  }],
  "action": { "default_popup": "popup.html" }
}
```

**Key permissions**: `storage` + `unlimitedStorage` lifts the 10MB cap on `chrome.storage.local` (critical for multi-hour meetings). `downloads` enables file export via `chrome.downloads.download()`. The `host_permissions` entry for `api.anthropic.com` enables CORS-free API calls from the service worker.

The **content script is the primary workhorse** — it runs as long as the Meet tab is open and doesn't suffer from the service worker's lifecycle limitations. The architecture follows a three-tier pattern:

```
Content Script (ISOLATED world)     Service Worker              Popup (ephemeral)
├─ MutationObserver on captions     ├─ Receives transcript      ├─ Reads state from storage
├─ In-memory transcript buffer      │  chunks via messages       ├─ Start/Stop controls
├─ Periodic flush (every 30s)       ├─ Updates badge ("REC")    ├─ Live preview via
│  via chrome.runtime.sendMessage   ├─ Handles file export      │  storage.onChanged
└─ Long-lived port to keep          ├─ Proxies Claude API       └─ Destroyed on close
   service worker alive             └─ Stores to storage.local
```

**The MV3 service worker terminates after ~30 seconds of inactivity** — the single biggest architectural difference from MV2. For an active recording session, keep it alive with a long-lived port connection:

```javascript
// content-script.js — keeps service worker alive during recording
const port = chrome.runtime.connect({ name: 'recording-session' });
const keepAlive = setInterval(() => port.postMessage({ type: 'PING' }), 25000);
// On recording stop: clearInterval(keepAlive); port.disconnect();
```

Content scripts in MV3 **cannot make authenticated cross-origin requests** — they inherit the web page's origin and are subject to CORS. All external API calls must go through the service worker. Content scripts also run in an isolated JavaScript world: they can read and modify the DOM but cannot access Google Meet's internal JavaScript objects.

## Open-source projects that work today and what they teach us

The most valuable reference implementations, ranked by maturity and recency:

**TranscripTonic** (github.com/vivek-nexus/transcriptonic) is the most actively maintained project — **171+ stars, 275 commits, Chrome Web Store published, last release February 2026**. It supports auto-mode (records all meetings automatically), manual mode, webhook integration, and multi-language captions. All processing happens locally in the browser with no data leaving the device unless a webhook is configured. It handles transcript recovery for interrupted sessions.

**Recall.ai's Chrome extension** (github.com/recallai/chrome-recording-transcription-extension) is the most architecturally modern — **TypeScript, MV3, Offscreen API for audio mixing**. Its `scrapingScript.ts` file contains the actual caption DOM parsing code and is the best single-file reference for understanding current selectors. The companion blog post at recall.ai/blog explains the architecture decisions.

**Recall.ai's headless bot** (github.com/recallai/google-meet-meeting-bot) demonstrates a Playwright-based approach — a containerized bot joins meetings, enables captions via `Shift+C`, scrapes the DOM with `setInterval`, and POSTs transcripts to an OpenAI summarizer. This is the most detailed source for current selector code including the `getSpeaker`/`getText` pattern shown earlier.

**Meet-Transcript-Saver** (github.com/Yaswanth-ampolu/Meet-Transcript-Saver) is the simplest implementation — a single `content.js` file with MutationObserver, multiple fallback selectors, and export to TXT/JSON/CSV/HTML. Good starting point for a minimal viable extension.

An older but instructive approach from a **Medium bookmarklet** by Ajay Prabhakar uses polling (`setInterval` at 5-second intervals) rather than MutationObserver, targeting `.iTTPOb` (caption text) and `.zs7s8d` (speaker name) classes. This polling approach is simpler but misses captions that appear and disappear between intervals.

## Speaker identification relies on fragile but functional DOM patterns

Google Meet displays speaker names as **bold label badges** in separate DOM elements from the caption text, nested within the same caption container `<div>`. The speaker badge uses classes like `.NWpY1d` or `.xoMHSc` — both have been observed in production code. The reliable extraction pattern is:

1. Find the caption container element (triggered by MutationObserver)
2. Query for the badge element within it to get the speaker name
3. Clone the container, remove the badge, and read remaining `textContent` for the caption

**The local user's captions display "You" as the speaker name** rather than the user's actual name. Extensions like sughodke/google-meet-transcripts handle this with a configurable speaker-name mapping: `{"You": "Michael", "Jim Halpert": "Jim"}`.

When multiple people speak simultaneously, Google Meet renders **separate caption containers per active speaker**. Attribution works correctly as long as the badge-to-text pairing within each container is maintained. However, Recall.ai explicitly warns that speaker labels "aren't embedded in a stable, machine-readable way. Parsing them still relies on fragile selectors or visual heuristics, making attribution lossy and error-prone."

There is **no observable DOM structure difference** between individual user captions and "Turn on captions for everyone" — the caption rendering mechanism is the same. The difference is only in who sees captions (the individual vs. all participants), not how they're rendered.

## Storage strategy and file export for transcript data

The optimal storage strategy uses a **hybrid in-memory buffer plus periodic flush** pattern:

**During recording**, accumulate transcript entries in a content-script JavaScript array. This avoids constant storage writes. Every **30 seconds**, flush the buffer to `chrome.storage.local` via message passing to the service worker. This balances performance (fast in-memory writes) with durability (regular persistence in case of crashes). Without `unlimitedStorage`, `chrome.storage.local` is capped at **10MB** — sufficient for roughly 5–10 hours of transcript. With the permission, storage is effectively unlimited.

For **IndexedDB**, it's available in both content scripts and service workers and handles much larger datasets with structured queries. However, the `unlimitedStorage` permission doesn't apply to IndexedDB — it's subject to browser quota management (up to ~60% of disk). Use IndexedDB only if you need binary data storage (audio recordings) or complex querying.

**File export** works best with `chrome.downloads.download()` using data URLs from the service worker:

```javascript
// service-worker.js
function exportTranscript(data, format) {
  let content, mimeType, extension;
  if (format === 'markdown') {
    content = data.entries.map(e => `**${e.speaker}** (${e.time}): ${e.text}`).join('\n\n');
    mimeType = 'text/markdown'; extension = 'md';
  } else if (format === 'json') {
    content = JSON.stringify(data, null, 2);
    mimeType = 'application/json'; extension = 'json';
  } else {
    content = data.entries.map(e => `[${e.time}] ${e.speaker}: ${e.text}`).join('\n');
    mimeType = 'text/plain'; extension = 'txt';
  }
  chrome.downloads.download({
    url: `data:${mimeType};charset=utf-8,${encodeURIComponent(content)}`,
    filename: `meet-transcript-${Date.now()}.${extension}`,
    saveAs: true
  });
}
```

Content scripts cannot use `chrome.downloads` directly — they must message the service worker. As a fallback, content scripts can create Blob URLs and trigger downloads via anchor element clicks.

## Meeting metadata extraction from the page and URL

The **meeting code** is trivially extracted from the URL: `window.location.pathname.replace('/', '')` yields the `abc-defg-hij` code, or use a regex: `/meet\.google\.com\/([a-z]{3}-[a-z]{4}-[a-z]{3})/`.

**Meeting title** extraction is more fragile — it lives in the top bar DOM and requires a class-name-dependent selector. TranscripTonic implements "meeting title detection" with multiple fallback selectors. There's no stable ARIA attribute for the meeting title.

**Participant names** can be extracted from the participant image container: `document.querySelectorAll('div.U04fid > img')` — each image has a `title` attribute containing the participant's name. This comes from a Google Codelab tutorial and is relatively stable.

**Meeting start detection** uses the appearance of the leave button: `button[aria-label*="Leave call"]`. **Meeting end** detection watches for that button disappearing or the leave banner appearing. Multiple extensions also watch for text content like "You've been admitted" or "You're the only one here" as join confirmation signals.

## Calling Claude from a Chrome extension bypasses CORS entirely

**The service worker is the correct place for Claude API calls.** With `"https://api.anthropic.com/*"` in `host_permissions`, the service worker makes `fetch()` calls without CORS restrictions — Chrome exempts extension-privileged contexts from CORS enforcement entirely. No preflight requests are sent, and the full response body is accessible regardless of what CORS headers the Anthropic API returns.

Content scripts **cannot** call the Anthropic API directly — they inherit the web page's origin and are blocked by CORS. The pattern is: content script sends transcript via `chrome.runtime.sendMessage()` → service worker calls Claude → service worker returns summary.

**Streaming is strongly recommended** for meeting summaries. Set `"stream": true` in the request body to receive Server-Sent Events. Each SSE chunk arriving resets the service worker's idle timer, preventing termination during long summarization. A typical 1-hour meeting transcript (~15,000–20,000 tokens) costs approximately **$0.07 with Claude Sonnet** or **$0.02 with Claude Haiku**.

For the service worker lifecycle during API calls, use the `waitUntil` keep-alive pattern:

```javascript
async function waitUntil(promise) {
  const keepAlive = setInterval(chrome.runtime.getPlatformInfo, 25000);
  try { return await promise; }
  finally { clearInterval(keepAlive); }
}
```

**API key security**: Never expose keys in content scripts. Store user-provided keys in `chrome.storage.local` (accessed only from the service worker). For production extensions, route through a backend proxy that holds the key server-side. `chrome.storage` has no built-in encryption — consider encrypting keys with the Web Crypto API (AES-GCM) before storing.

For long transcripts exceeding token limits, chunk the transcript at paragraph boundaries, summarize each chunk independently, then run a final synthesis pass combining chunk summaries into a coherent whole.

## Popup UI communicates through storage, not direct connections

The popup is **destroyed the moment it loses focus** — never store important state in popup JavaScript variables. On open, read current state from `chrome.storage.session`; for live updates, listen to `chrome.storage.onChanged`:

```javascript
// popup.js
document.addEventListener('DOMContentLoaded', async () => {
  const { isRecording } = await chrome.storage.session.get(['isRecording']);
  updateUI(isRecording);
  
  chrome.storage.onChanged.addListener((changes, area) => {
    if (area === 'session' && changes.transcriptPreview) {
      document.getElementById('preview').textContent = changes.transcriptPreview.newValue;
    }
  });
});
```

**Badge text** provides persistent visual feedback even when the popup is closed:

```javascript
// service-worker.js
chrome.action.setBadgeText({ text: isRecording ? 'REC' : '' });
chrome.action.setBadgeBackgroundColor({ color: '#e74c3c' });
```

For start/stop controls, the popup sends messages directly to the content script: `chrome.tabs.sendMessage(tab.id, { type: 'START_RECORDING' })`. Export triggers go to the service worker: `chrome.runtime.sendMessage({ type: 'EXPORT_TRANSCRIPT' })`.

## Critical gotchas that will break your extension

**DOM selector breakage is the #1 risk.** Google ships Meet UI updates continuously through their rapid release process. TranscripTonic has released patches specifically to "Handle new meeting lobby UI" after selectors broke. The mitigation hierarchy: use `[aria-live]` and `aria-label` attributes first (accessibility-mandated, rarely removed), fall back to structural selectors (`div > span:first-child`), and avoid class names when possible. Centralize all selectors in one file for rapid hot-patching.

**Memory leaks compound over long meetings.** Undisconnected MutationObservers keep observed nodes in memory even after DOM removal. Unbounded segment buffers grow linearly — Recall.ai recommends a hard cap of **10,000 segments** and periodic flushing to storage. `console.log` statements can retain object references for the entire page lifetime (documented Mozilla bug). Always call `observer.disconnect()` on cleanup.

**The service worker dying mid-meeting is architecturally inevitable in MV3.** Global variables are lost on termination. The content script must maintain a long-lived port connection with keep-alive pings every 25 seconds. Use `chrome.alarms` for periodic tasks. Store all state in `chrome.storage`, never in service worker memory. TranscripTonic implements a "Recover last meeting" feature specifically for handling unexpected disconnections.

**Caption text is lossy by nature.** Google's speech recognition "paraphrases aggressively, truncates long phrases, and omits punctuation." Recall.ai's testing found caption-based transcription achieves ~**82% accuracy** vs. ~85.7% for Google's post-meeting API transcripts. Captions are transient — they appear briefly then disappear from the DOM. If the MutationObserver callback is slow, text can be missed entirely.

**Google Meet warns users about extensions** with a banner: "You have extensions installed that may affect the quality of your call." This is triggered by content script injection and cannot be suppressed. For headless bot approaches, frequent automated joins trigger CAPTCHAs and 2FA prompts — Google actively discourages bot access to Meet.

**Tab visibility throttling** can degrade performance. Chrome throttles JavaScript execution in background tabs, potentially causing the MutationObserver to miss rapid caption updates. The `tabCapture` API (for audio-based approaches) requires the Meet tab to remain active and visible. Content-script-based caption scraping is more resilient to tab backgrounding than audio capture, but users should be advised to keep the Meet tab active.

## Conclusion

The most resilient architecture uses **ARIA attributes as primary selectors** with class-name fallbacks, a **content script with MutationObserver** watching `childList` + `characterData` + `subtree`, **30-second periodic flushes** to `chrome.storage.local` with `unlimitedStorage`, and the **service worker solely as a coordinator** for badge updates, file exports, and Claude API proxy calls. The open-source TranscripTonic and Recall.ai projects provide production-tested reference code. The fundamental tension is that this entire approach depends on reverse-engineering an unstable DOM — there is no supported API. Build with the expectation that selectors will break, centralize them for quick fixes, and implement transcript recovery for resilience. Google's February 2025 caption scrollback and June 2025 customizable caption styling are the most recent DOM-affecting changes to watch for.
